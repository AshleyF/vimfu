# VimFu Content System

This directory is the **single source of truth** for all VimFu instructional
content. From here we render:

- a **website** (the *reference* — fast, searchable, hosts videos);
- a **book** (the *deep dive* — comprehensive, developer-focused, paper-friendly);
- intermediate **markdown** for human review;
- **InDesign Tagged Text** for layout;
- **SVG screenshots** of worked examples (color for web, B&W for print).

One set of source files. Many output formats. No content duplicated, no
content drift.

---

## 1. Goal — one source, two products

We're not writing a book. We're not writing a website. We're authoring a
**content graph** that produces both, with deep cross-linking between
them.

| Surface | Role | Tone | Format |
|---------|------|------|--------|
| **Website** | reference; hosts videos | concise, look-up oriented | HTML + embedded YouTube |
| **Book**    | comprehensive guide | developer-focused, explains *why* | print (InDesign → PDF) |

Each topic is the same source, but renderers select the right pieces:

- The site embeds short videos (we already have ~250 on YouTube).
- The book replaces those with **QR codes** that bounce the reader to
  the matching site URL — every code → page; every page → codes.
- The book includes longer **`internals`** prose explaining how Vim
  *actually works* (state, registers, motion classes, FSM).
- The site has a **`buy-prompt`** upsell pointing at the book.

Both surfaces share the same diagrams, screenshots, key tables, and
worked examples — rendered to whatever the destination needs.

### Voice

The book is for developers. We don't say "press d to delete" — we
explain that `d` is an *operator* that consumes a *motion or text
object*, that the result lands in the *unnamed register* and the
*numbered register `"1`*, and that this is why `dd`, `dw`, `d/foo`,
and `daw` are all the same thing under the hood.

We acknowledge warts. `Y` is broken symmetry. `cw` doesn't behave like
`dw`. `%` is overloaded. The book explains *why*, then shows the way
through.

## 3. Content model in 60 seconds

```
parts/<NN>-<part-slug>/<NN>-<topic-slug>.json   →  one Topic
                          │
                          ├─ id          "registers.unnamed"   (stable across renames)
                          ├─ title       "The Unnamed Register"
                          ├─ subtitle    short tagline
                          ├─ summary     1–2 sentence reference description
                          ├─ keys        ["yy", "p", "P", ...]
                          ├─ lessons     [85, 86]              (YouTube short numbers)
                          ├─ see_also    [<topic-id>, ...]     (cross-refs by id)
                          └─ blocks      [ ... rendered in order ... ]
```

### Block types (the building bricks)

See [`Schema.md`](Schema.md) for the full spec.

| Block         | Renders as |
|---------------|------------|
| `heading`     | `<h2>`/`##`/`Heading2` style |
| `prose`       | a paragraph; supports `{key:X}`, `[label](#topic.id)`, ``code``, **bold**, *italic*, ``` ``` fenced code |
| `keys`        | a key-and-note table |
| `table`       | an inline table |
| `tip`         | a yellow callout box |
| `internals`   | a "Under the Hood" callout — book-emphasized |
| `embed`       | site shows a YouTube iframe; book skips |
| `qr`          | book shows a QR code → site URL; site skips |
| `buy-prompt`  | site shows an upsell to the book; book skips |
| `example`     | `{ "ref": "<example-id>" }` — see §4 |
| `screenshot`  | a single inline frame |
| `divider`     | horizontal rule / `* * *` |

### Audience tagging — book vs. web vs. both

Topics and individual blocks can carry an `"audience"` field:

| Value    | Where it appears |
|----------|------------------|
| `"both"` (default) | All renderers |
| `"web"`  | HTML only (skipped by InDesign / book) |
| `"book"` | InDesign only (skipped by HTML / web) |
| `"all"`  | Same as `both` |

The Markdown renderer always emits everything (it's the review format).

Use `audience: "book"` to author *fluff* — history, philosophy, longer essays,
asides — that belongs in the printed book but would clutter the web reference.
Use `audience: "web"` for things that only make sense online (interactive
widgets, "back to top" links, etc.) Most content stays at the default
`both` and renders identically in all formats.

```jsonc
// whole-topic book-only
{ "id": "foundations.adm-3a", "audience": "book", "blocks": [...] }

// single book-only sidebar inside an otherwise-shared topic
{ "type": "internals", "audience": "book",
  "title": "Why hjkl, specifically?",
  "text": "..." }
```

Renderer audience defaults:

| File                   | `AUDIENCE` |
|------------------------|------------|
| `render_html.py`       | `"web"`    |
| `render_indesign.py`   | `"book"`   |
| `render_markdown.py`   | `"all"`    |

### Inline markup (works inside `prose`, `summary`, `tip`, `internals.text`, …)

| Markup | Renders as |
|--------|------------|
| `{key:X}` | `<kbd>X</kbd>` (HTML), `` `X` `` (Markdown), `<CharStyle:Key>X<CharStyle:>` (InDesign) |
| `[label](#topic.id)` | resolved to a relative link to that topic's output file |
| `` `code` `` | inline code |
| `*em*`, `**strong**` | italic / bold |
| ` ```...``` ` | fenced code block (pre/code, CodeBlock paragraph style) |

## 4. Worked examples and screenshots

Every meaningful topic gets a **worked example** — a short sequence of
terminal screenshots showing a real edit, with caption + narration per
frame. Screenshots are rendered as **self-contained SVG** in two themes:

- `color` — the original terminal palette (used by web, markdown).
- `bw`    — book-friendly grayscale (used by InDesign).

### Authoring an example

We keep all examples in **one Python file** — `build_examples.py` —
because hand-authoring 100+ JSON files with run-arrays is brutal. The
script defines small helpers (`E`, `F`, `hl`) and one `add(...)` call
per example. Running the script writes each example as a JSON file in
`content/examples/`.

A frame uses the **compact DSL**: a list of plain strings, a `(row, col)`
cursor, optional `highlights` (color/bold overrides for spans of cells),
and optional `status` / `mode_line` shortcuts. The compact DSL is
expanded into the canonical [Frame Format](../brainstorming/frame_format.md)
that the SVG renderer consumes.

```python
add(E("editing.delete",
    "x dw dd",
    "Three deletes, three scopes.",
    topic="editing.delete",
    frames=[
      F(["the quick brown fox"], cursor=(0,4),
        caption="Cursor on 'q'."),
      F(["the brown fox"], cursor=(0,4), keys="dw",
        caption="dw — through next word boundary."),
      F([""], cursor=(0,0), keys="dd",
        caption="dd — entire line.",
        narration="d is the operator; x is sugar for dl."),
    ]))
```

Embed the example in any topic with one line:

```json
{ "type": "example", "ref": "editing.delete" }
```

### Two ways to source frames

`render_screenshots.py` accepts three frame sources, in priority order:

1. **`frame`** — full Frame JSON inline (rare; for bespoke spot illustrations).
2. **`compact`** — compact-DSL inline, expanded by `lib/compact_frame.py`.
   This is what `build_examples.py` produces.
3. **`source.lesson` + `source.select`** — replays an existing
   curriculum lesson via `shellpilot/capture_frames.py`, captures pyte's
   screen buffer with full color, caches it under
   `output/screenshots/_captured/`, then picks the indices listed.
   We have **749** curriculum shorts available as a frame source.

### Why SVG, why two themes

SVGs are vector — they scale crisply on web *and* in print without
re-rendering. The renderer assigns semantic CSS classes
(`.bg0/.bg1/...`, `.fg0/.fg1/...`) and embeds a `<style>` block per file,
so:

- A new theme is just another CSS template (high-contrast, sepia,
  brand-colored, etc.).
- Each SVG is **self-contained** — drag-and-drop into anything that
  renders SVG and the colors come along.

## 5. Rendering pipeline

Four scripts. They can run in any order, but the screenshot pipeline
must run before the topic renderers if you want the screenshots picked
up.

```
content/
  parts/*.json   ─┐
                  │  render_markdown.py    ──► output/markdown/
  examples/*.json─┤  render_html.py        ──► output/html/  + style.css
                  │  render_indesign.py    ──► output/indesign/  + README.txt
                  │
                  └─ render_screenshots.py ──► output/screenshots/<id>/*.svg
                       (replays curriculum
                        lessons via shellpilot
                        if needed)
```

### Build commands

From the repo root:

```bash
# 1) Generate examples/*.json from build_examples.py
python content/build_examples.py
#    or filter:  python content/build_examples.py grammar

# 2) Generate screenshots (color + bw SVG per frame)
python content/render_screenshots.py
#    or one example:  python content/render_screenshots.py grammar.dw-then-dot

# 3) Generate per-topic QR codes (encode vimfubook.com URLs)
python content/render_qrcodes.py
#    or staging:  python content/render_qrcodes.py --base-url https://staging.vimfubook.com

# 4) Generate the three text outputs (any subset, in any order)
python content/render_markdown.py
python content/render_html.py
python content/render_indesign.py

# 5) Copy the simulator runtime into output/html/sim/ for "Practice" links
python content/copy_sim.py

# 6) Build the print PDF (book.pdf) — KDP-ready
python content/render_latex.py     # writes content/output/latex/*.tex
python content/build_pdf.py        # SVG → PDF (Inkscape) → xelatex → makeindex → xelatex × 2

# 7) Publish the website to docs/ (GitHub Pages)
python content/deploy_site.py      # re-renders, bundles sim, syncs docs/
```

### Publishing the website (`docs/`)

**Renders go to `content/output/html/`. Publishing goes to `docs/`.**
GitHub Pages is configured to serve from `main` branch `/docs` folder
(repo Settings → Pages), so anything not under `docs/` is invisible to
the live site.

`content/deploy_site.py` is the canonical "publish" entry point. It:

1. Re-runs `render_redirects.py`, `render_errata.py`, `render_html.py`
   (so `content/output/html/` is fresh).
2. Builds the single-file bundled simulator via `sim/build.py` →
   `book/output/sim/index.html` (esbuild minifies all the sim JS/CSS
   into one HTML file so GitHub Pages can serve it without a
   `node_modules`).
3. Wipes `docs/` (preserving `CNAME` and `.gitkeep`).
4. Copies `content/output/html/` → `docs/`.
5. Replaces `docs/sim/` with the bundled single-file sim.
6. Drops a `.nojekyll` marker so Pages serves underscore-prefixed paths.

Then **commit `docs/` and push** to publish.

```powershell
python content/deploy_site.py            # full pipeline
python content/deploy_site.py --skip-render  # if you already ran renders
python content/deploy_site.py --skip-sim     # if you didn't change sim/
```

Do **not** edit files inside `docs/` by hand — they're regenerated.
Do **not** publish from `content/output/html/` — that's a build
artifact directory, not what GitHub Pages serves.

### Print PDF pipeline (KDP)

`render_latex.py` emits `content/output/latex/book.tex` + per-part files.
`build_pdf.py` then runs:

1. **SVG → PDF** for every screenshot/QR (~800 files). Uses Inkscape's
   `--shell` mode for batch throughput. If Inkscape is missing or
   broken, the build aborts with `ERROR: 'inkscape' not on PATH`.
2. **xelatex → makeindex → xelatex × 2** to resolve refs + the back-of-book
   index.

Inspect `content/output/latex/book.log` for overfull boxes and
"ignored error" lines.

#### Required tools

- **Inkscape 1.x** on `PATH` (or set `$env:PATH += ';C:\Program Files\Inkscape\bin'`).
  Append, never prepend — Inkscape's `bin/` ships a `python.exe` shim
  that hijacks system Python if it comes first.
- **MiKTeX** (xelatex + makeindex) with the Libertinus, TikZ, tcolorbox,
  microtype, longtable, and newunicodechar packages.

#### KDP layout constants (7.5×9.25, 301–500 page bucket — book is ~393 pages)

- **Trim size**: 7.5" × 9.25" (KDP "B6+" / non-standard, larger than 6×9).
- **Inner (gutter) ≥ 0.625"** for 301–500 pages — we use **0.875"** for generous headroom.
- Outer/top/bottom ≥ 0.25" minimum, 0.5" recommended — we use **0.625"** outer, **0.75"** top/bottom.
- `twoside,openany`. Defined in `render_latex.py` `\geometry{...}` (paperwidth=7.5in, paperheight=9.25in).
- **Watch the page-count buckets:** if the book ever grows past 500 the
  minimum gutter jumps to 0.875" (KDP table: 24–150=0.375, 151–300=0.5,
  301–500=0.625, 501–700=0.875, 701–828=1.0").

#### PDF-page parity must match physical-binding parity (KDP gutter check)

KDP's automated preflight reads the **PDF page number** and assumes
PDF page 1 = recto (right-hand). It then checks each page's inner
margin (left on odd PDF pages, right on even PDF pages).

LaTeX assigns `oddsidemargin` (inner = 0.875") to its **own** odd
pages and `evensidemargin` (outer = 0.625") to its own even pages.
If the frontmatter has an **odd** number of PDF pages, LaTeX page 1
of the mainmatter (odd → LaTeX recto, inner on left) lands on an
**even** PDF page — and KDP, looking at the right side as the gutter,
sees only 0.625" and flags the entire mainmatter as
"insufficient gutter."

The fix is in `render_latex.py`:

```latex
\renewcommand\mainmatter{\cleardoublepage\@mainmattertrue\pagenumbering{arabic}}
```

`\cleardoublepage` (instead of plain `\clearpage`) forces a blank
verso when needed so the mainmatter always starts on a recto **and**
on an odd PDF page. Do not change this back to `\clearpage` even if
the resulting blank verso looks like a "stranded blank page" — it is
load-bearing for KDP gutter compliance. `\frontmatter` and
`\backmatter` deliberately still use `\clearpage` because openany
doesn't need them aligned.

Symptom if this regresses: KDP flags 10–30 specific mainmatter pages
with "insufficient gutter" — typically pages where the (now-outer)
margin dips slightly under 0.625" because of italic overshoot, key
pills, or callout rules. The pages themselves look fine in the PDF;
KDP is just reading the wrong side as the gutter.

#### Inkscape PDF export flags

`build_pdf.py` passes `export-text-to-path:true` so every text element
in a screenshot SVG becomes vector outlines in the PDF. Without this
flag, KDP's preflight rejects the screenshots as
**"fonts not properly embedded"** — Inkscape's default PDF exporter
embeds Cascadia/JetBrains subset fonts but doesn't always include a
full ToUnicode mapping. Vector paths sidestep the embedding problem
and are pixel-identical to the rasterized text.

#### `\XeTeXgenerateactualtext=1` (KDP non-printable workaround)

The preamble sets `\XeTeXgenerateactualtext=1`. This forces xelatex to
wrap every glyph run in a marked-content block with `/ActualText` so
PDF text extraction reads the original Unicode regardless of the
font's ToUnicode CMap.

Libertinus's CMap as generated by `xdvipdfmx` maps several glyph IDs
(stylistic/ligature variants xdvipdfmx couldn't reverse-map) to C0
control chars: **U+0007, U+0009, U+001A, U+001B, U+001D**. When KDP
extracts page text it sees these control bytes and reports
**"non-printable markup that was removed"** with little marker icons
in the previewer. ActualText overrides the CMap and resolves the
issue without changing the rendered glyphs.

#### Known build gotchas

- **Inkscape PATH on Windows: APPEND, never PREPEND** (see above).
- **Inkscape can be silently broken after a Windows update** — symptom is
  `OSError [WinError 216] not compatible with this version of Windows`,
  and `inkscape.exe` / `inkscape.com` shrink to ~407 KB. Fix:
  `winget uninstall --id Inkscape.Inkscape` then
  `winget install --id Inkscape.Inkscape` (a reinstall replaces the
  damaged binaries — a plain upgrade does not).
- **`\key{...}` is TikZ-based** → fragile inside `\index{}`. Always emit
  `\protect\key{...}` for the display half of an index entry
  (`\index{Keys!h@\protect\key{h}}`). Without `\protect`, TikZ's
  catcode-deactivation runs during `.idx` write inside `\textit{}` /
  other fragile contexts and crashes with
  `Undefined control sequence \tikz@deactivat...`. See
  `_index_key_entry()` in `render_latex.py`.
- **Multi-key sequences in `\index{}`** (e.g. `Ctrl-W Ctrl-W`) must be
  split on spaces into separate `\key{}` pills — one wide pill
  overflows the narrow index columns. `_index_key_entry()` and
  `_split_key_for_wrap()` both do this.
- **Non-printable chars trip KDP's preflight.** xelatex emits U+0000 for
  missing glyphs, which KDP flags. Map any unmappable Unicode via
  `\newunicodechar` in the preamble. The known offender is **✓**
  (U+2713) in Libertinus Serif — mapped to `\ensuremath{\checkmark}`.
- **No NBSPs in screenshot SVGs.** `lib/svg.py` must NOT replace spaces
  with U+00A0. Each `<tspan>` char already has an explicit `x`
  attribute, so use `xml:space="preserve"` and emit literal spaces.
  NBSPs leak into the PDF text layer and KDP rejects them.
- **`book/marketing/_scan_pdf.py`** scans `book.pdf` for NBSPs, NULs,
  control chars, PUA, soft hyphen, BOM. Run it after every PDF build
  before uploading to KDP.

### The simulator integration

Every topic page in the website gets a **▶ Practice in the simulator**
link in its top nav. The link deep-links into a copy of `sim/` that
ships at `output/html/sim/`. Supported query params (handled in
`sim/index.html`):

| Param      | Effect |
|------------|--------|
| `?file=NAME&content=TEXT` | Seed/replace a file in the sim's VFS, then run `nvim NAME` at the shell. |
| `?cmd=CMD` | Type `CMD` at the shell prompt and execute it. |
| `?keys=KS` | Feed literal keystrokes once the session is ready. URL-encode special keys. |
| `?reset`   | Wipe the sim's localStorage VFS so seed files are re-created. |

`render_html.py` builds the per-topic query string in
`_practice_query()`. For Vim topics it seeds `practice.txt` with the
first wired example's starting buffer. For tmux topics (anything in
`parts/22-tmux/`) it sends `?cmd=tmux` so practice drops you straight
into a tmux session. Override per-topic by giving the topic an
explicit `practice` field once we add support — for now, examples and
the part name decide.

## 6. Cross-referencing and IDs

IDs are **stable forever**. They show up in URLs, in QR codes, in
cross-references between topics, and in `topic` fields on examples.

Format: `<part-slug>.<topic-slug>` — both lowercase, hyphenated,
ASCII-only. Examples: `editing.yank-put`, `registers.unnamed`,
`patterns.cgn-dot`.

When you cross-reference inside prose, use:

```
... see [the grammar](#text-objects.grammar) for the full ...
```

Renderers resolve the `#topic.id` against the topic index and produce
correct relative links per output format.

If an id is renamed, every renderer turns broken links into a visible
warning so they're easy to find.

## 7. Lessons, YouTube videos, and QR codes

### Video integration

We have **685 short lessons** indexed under `curriculum/shorts/*.json`
(plus longs); ~292 are published to YouTube with a real `youtube.videoId`.
The rest ship metadata only — they'll resolve automatically once
uploaded. The video index lives in `content/lib/videos.py` and is loaded
by every renderer.

Each topic carries a `lessons: [N, ...]` array of lesson numbers. The
renderers use this in two ways:

1. **Explicit `embed` block** — `{"type": "embed", "lesson": 31, "caption": "..."}`
   places one video at a precise spot in the topic body.
2. **Auto "Watch" section** — every lesson in `lessons[]` *not* already
   covered by an explicit `embed` block is rendered into an
   end-of-topic "Watch" gallery. So adding a lesson number to `lessons[]`
   is enough for it to show up; explicit `embed` blocks are only for
   anchoring a video to a specific spot in the prose.

Per output:

| Surface  | `embed` block        | Auto "Watch" section |
|----------|----------------------|----------------------|
| HTML     | 9:16 YouTube iframe  | grid of 9:16 iframes + lesson titles |
| Markdown | YouTube link with title | bulleted list of lesson links |
| InDesign | inline-graphic placeholder + URL hint | "Watch Online" body paragraphs |

Unpublished lessons render as visible placeholders with the title
intact, so they're easy to spot and update when uploads happen.

### QR codes

The book is `vimfubook.com`'s ambassador. Every topic ends with a `qr`
block that points at the same topic's website page. The pipeline:

```
content/render_qrcodes.py
   → output/qrcodes/<part_dir>/<file_stem>.svg
   each encoding: https://vimfubook.com/<part_dir>/<file_stem>
```

QR codes are SVG (vector — print at any size). The base URL is
configurable: `python content/render_qrcodes.py --base-url https://staging.vimfubook.com`.

The renderers use the QR codes like this:

| Surface  | `qr` block |
|----------|------------|
| HTML     | inline `<img src="../qrcodes/...svg">` next to a "Scan or visit ..." link |
| InDesign | placeholder paragraph hinting at the SVG path + the encoded URL — designer drops in the file |
| Markdown | text-only link (no need for a QR code on a screen) |

`content/lib/qr.py` exposes `qr_svg(text)` and `topic_qr_svg(topic)`
for any other tool that needs to build a QR.

### Build order with videos and QR

```bash
python content/build_examples.py       # examples/*.json
python content/render_screenshots.py   # output/screenshots/
python content/render_qrcodes.py       # output/qrcodes/
python content/render_markdown.py
python content/render_html.py
python content/render_indesign.py
```

Renderers tolerate missing artifacts (screenshots or QR SVGs) — they'll
emit broken-link references instead of crashing. So a developer working
only on prose can skip the asset steps.

## 8. Authoring workflow (the loop)

For a brand-new topic:

1. Pick the right part directory under `parts/`.
2. Create `<NN>-<topic-slug>.json` with the full topic schema (see
   [`Schema.md`](Schema.md), and copy a similar topic for shape).
3. Add the topic id to its part's outline in [`Outline.md`](Outline.md).
4. If the topic deserves a worked example, add one entry in
   `build_examples.py`. If it deserves more than one, add more.
5. Reference any examples with `{ "type": "example", "ref": "..." }`
   blocks inside the topic's `blocks` array.
6. Run the four render commands; review the markdown output for
   sanity, the HTML for visual layout, and Tagged Text for designer
   import.

For an *edit* to an existing topic — same loop minus steps 1–3.

### Style rules

- **Developer voice.** Explain the *mechanism*, not just the keystroke.
- **Acknowledge warts.** Be honest about Vim's broken symmetries — readers
  recognize them.
- **Reference, don't duplicate.** Cross-link to the canonical topic
  rather than re-explaining a concept.
- **Every topic ends with `qr` then `buy-prompt`** so both surfaces have
  the right cross-promotion.

See [`Authoring.md`](Authoring.md) for the longer style guide.

## 9. Renderer internals (when you need to extend)

Each renderer follows the same shape:

1. `load_topics()` reads every `parts/**/*.json`.
2. `build_index(topics)` builds an id-→ file-stem index for cross-refs.
3. `load_examples()` reads every `examples/*.json` into a dict keyed by id.
4. `render_inline(text, …)` rewrites `{key:…}`, `[label](#id)`,
    inline code/bold/em, and (in HTML/InDesign) splits ``` ``` fenced blocks.
5. `render_block(block, …)` returns the formatted output for one block.
6. `render_topic(t, index, examples)` joins blocks for one topic.
7. Walk all topics, write per-part indexes, write a root index, write
   any shared assets (CSS, designer README).

To add a new block type:

1. Add the spec in `Schema.md`.
2. Handle it in **all three** renderers (`render_markdown.py`,
   `render_html.py`, `render_indesign.py`).
3. Add corresponding CSS / Tagged Text paragraph styles if needed.

To add a new theme to the SVG renderer:

1. Add a CSS-builder function alongside `_color_css` and `_bw_css` in
   `lib/svg.py`.
2. Wire the theme name through `frame_to_svg(frame, theme="...")`.
3. Update `render_screenshots.py` to emit the new theme files.

## 10. The screenshot library — `lib/`

### `lib/svg.py`

```python
from lib.svg import frame_to_svg

svg_color = frame_to_svg(frame, theme="color")
svg_bw    = frame_to_svg(frame, theme="bw")
```

- Discovers each frame's palette and assigns numbered classes.
- Embeds a `<style>` block so the SVG is fully self-contained.
- Handles the cursor as a separate overlay group — never disturbs the
  underlying text styling.

### `lib/compact_frame.py`

```python
from lib.compact_frame import expand
frame = expand({
  "rows": 8, "cols": 40,
  "lines": ["foo bar", "baz", "~", "~", "name.txt   1,1   All"],
  "cursor": [0, 4],
  "statusRow": 4,
  "highlights": [{"row": 0, "col": 4, "len": 3, "fg": "ff8700", "b": True}],
})
```

- Pads short lines to the column count.
- Auto-colors `~` rows with Vim's classic blue.
- Auto-applies reverse video to a `statusRow`.
- Run-length encodes per-row to produce canonical Frame JSON.

## 11. Curriculum lesson capture (advanced)

If you want a screenshot from a *real* lesson rather than authoring one
by hand, set up the example with a `source` block:

```json
{
  "format": "vimfu-content-example",
  "id": "grammar.real-capture",
  "title": "...",
  "topic": "...",
  "source": {
    "lesson": "curriculum/shorts/0085_the_vim_grammar.json",
    "select": [0, 7, 14, 21]
  },
  "frames": [
    { "caption": "Initial state." },
    { "caption": "After dw." },
    ...
  ]
}
```

`render_screenshots.py` will:

1. Lazy-import `shellpilot/capture_frames.py`.
2. Replay the lesson headlessly, capturing pyte's screen buffer at
   every visible step.
3. Cache the capture in `output/screenshots/_captured/<lesson>.json`.
4. Pick the frames at indices `select[0]`, `select[1]`, …
5. Render each into `frame_NN.color.svg` and `frame_NN.bw.svg`.

The number of `select` indices must match the number of `frames`
entries; they pair up one-to-one (caption / narration are taken from
the example file, not the captured frames).
