Counted Macros

100@a — run a macro a hundred times. Or until it errors out.

id: macros.counted · part: macros · keys: {n}@a · lessons: 130

Counts work on macros. 100@a runs macro a up to 100 times; if any iteration aborts (search miss, end of file), playback stops cleanly.

A macro that does its work and moves to the next item is a loop. {n}@a runs that loop n times — but it stops as soon as the macro errors. So you can confidently say "do this 1000 times" knowing it'll stop when there's nothing left.

🔧 Under the Hood — Why Errors Are Helpful Here

Vim's macro playback halts on an error. Search misses, edge-of-buffer motions, failed operations — anything that beeps interactively also aborts a macro mid-stream.

This is a feature. If your macro processes a list and you don't know how long it is, write the macro to fail gracefully when it runs out of work. Then run it with a count larger than you'll ever need:

  • Macro that searches forward for TODO, edits, moves to next: 1000@a. When /TODO finds no more matches, the macro aborts cleanly.
  • Macro that processes a CSV row by jumping to columns: 100@a. When the cursor falls off the end of the file, the next motion errors and playback stops.

Big counts don't cost much — Vim doesn't run the macro 1000 times unless there's work for it to do.

Tip. If you'd rather suppress errors and keep playing, use 100@a inside an :silent! or wrap with try/catch in scripting. For interactive work, the abort-on-error behavior is what you want.
🔧 Under the Hood — Counts re-invoke macros, they don't transform them

5@a runs the recorded stream five times in sequence; it does not multiply counts inside the recording. Side effects (cursor position, register state, search state) carry over from one invocation to the next. That's how 5@a plus a motion-at-end-of-recording produces a sweep instead of doing the same thing five times in place.
