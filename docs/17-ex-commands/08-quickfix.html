Quickfix and Location Lists

A typed list of file:line:message — the spine of grep, make, and lint workflows.

id: ex.quickfix · part: ex-commands · keys: :make, :grep, :vimgrep, :copen, :cnext, :cprev, :cclose

:make, :grep, and :vimgrep populate the quickfix list. :copen shows it; :cnext / :cprev step through entries. The location list is the per-window twin.

The quickfix list is Vim's universal results pane. Compile errors, grep hits, lint warnings, test failures — they all funnel into the same list with the same navigation keys. Once you wire your tools through it, every command-line workflow becomes a structured editing session.

Populate the list

Command Effect
:make Run 'makeprg' (default make), capture errors via 'errorformat'
:make test Pass arguments to the make program
:grep PATTERN [PATH] Run external grep ('grepprg'), populate from results
:vimgrep /PATTERN/ FILES Run Vim's internal grep (no shell)
:vimgrep /TODO/ **/*.py Recursively find TODO in Python files
:cexpr system('rg --vimgrep PATTERN') Inject any tool's output via 'errorformat'

Navigate the list

Tip. In the quickfix window, Enter opens the entry under the cursor. p previews it without leaving. :colder restores a previous quickfix list — Vim keeps a stack of them.

Location list — same idea, scoped to a window

Replace every c-prefixed command with l: :lmake, :lgrep, :lopen, :lnext, :lprev. The list lives on the window, not the editor, so two splits can run two greps without stepping on each other.

🔧 Under the Hood — Quickfix is a list of typed entries

Internally each quickfix entry is { filename, lnum, col, type, text, valid }. Population functions (:make, :grep, :cexpr) convert tool output into that struct via the 'errorformat' scanf-like parser. The quickfix window is a buffer rendered from that list — that's why you can't really edit it (the renderer overwrites your changes). Navigation commands index into the list, open the entry's file (in the current window), and place the cursor at (lnum, col).

🔧 Under the Hood — Location lists are per-window quickfix

Vim keeps two parallel data structures: one global quickfix list (shared by all windows) and one location list per window. They use identical entry structure, identical commands modulo prefix, identical errorformat parsing. The duplication is intentional: it lets you run a project-wide search in the quickfix list while simultaneously running a file-local search in the location list of one window, without overwrites. The window-tied lifetime also means closing the window discards its location list.

Tip. Set :set grepprg=rg --vimgrep to use ripgrep — it's massively faster than grep and produces output matching Vim's default 'grepformat'.
