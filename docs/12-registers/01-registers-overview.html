Registers

A dictionary of named clipboards built from one primitive

id: registers.overview · part: registers · keys: "a, "A, "0, "-, "+, "*, "/, ":, "., "%, "#, "=, "_, p, P, y, d · lessons: 119, 121, 125

A register is a named slot. Yank/delete writes; put reads. The unnamed, named, numbered, clipboard, and special registers are all the same thing — different keys.

Most editors have one clipboard. Vim has fifty-something — and they're all the same thing. Once you see the model, every command that touches yanked or deleted text becomes obvious. This is the single most leveraged piece of Vim's design to understand.

The model

A register is a named slot. Each register holds a piece of text plus a type (charwise, linewise, or blockwise). Vim has a fixed set of registers, identified by a single character — a–z, plus numbers, plus a handful of special punctuation. To address one, type " followed by the register name.

That's it. "a selects register a for the next operation. yy writes a line into it. p reads from the most recently selected register and inserts. The whole register interface is built from those three pieces: select, write, read.

Tip. If you don't say "a, Vim picks for you — the unnamed register "", which is the default destination of every yank and delete. So plain yy p works because both ops use the unnamed register implicitly.

Looking inside

Run :reg to see every register's contents. This is one of the most useful debugging commands in Vim — most "why did that paste do that" mysteries get solved by reading :reg.

The full set

🔧 Under the Hood — Registers Are a Dictionary

Internally, registers are a single key→(text, type) dictionary. The lowercase letters give you 26 slots you control. The uppercase forms aren't extra slots — they're append aliases for the same lowercase slot ("A writes to register a in append mode). The numbered registers shift on each delete: when a new big delete arrives, it becomes 1, the old 1 becomes 2, and so on, dropping 9. The yank register 0 is special — it only updates on yank, not delete, which is why "0p reliably gives you the last thing you yanked even after a few deletes have moved through.

Special registers like "/, ":, "., "%, "# are read by Vim itself for searches, the last command, the last insert, the filenames. They are the same data structure as the clipboard — Vim just exposes its internal state through the register namespace.

The black-hole register "_ is a convenient null sink: writes go nowhere. This is the canonical way to delete without disturbing the unnamed register, e.g. "_dd to delete a line without overwriting your last yank.

🔧 Under the Hood — The Type Field

Every register stores not just text but a type: char (charwise), line (linewise), or block (blockwise). The type is set by the operation that wrote the register: yy writes linewise; yw writes charwise; Ctrl-V-then-y writes blockwise.

The type changes how p behaves. A linewise put inserts the text on its own line(s), below the current line. A charwise put inserts inline, after the cursor. A blockwise put inserts a rectangular block to the right of the cursor. So the same key, p, dispatches on the type of the current register.

This is why it matters that x (a charwise delete) overwrites the unnamed register's type. If your last yank was yy (linewise), then a single x flips the unnamed register to charwise containing one character, and your next p pastes one character inline rather than a whole line. The fix: yank the line into a named register ("ayy) before you start deleting.

🔧 Under the Hood — Macros Are Just Register Contents

Recording a macro into register a writes the keystrokes — as text — into register a. Playing it back reads the text and feeds it to the input buffer.

This means you can:

  • Paste a macro: "ap shows the keystrokes as plain text on the page.
  • Edit a macro by hand: paste, edit, then yank back into the register: "ap, edit the line, 0"ay$ to put the modified text into register a.
  • Build a macro from text: type the keystrokes you want, yank them into a register, then @ plays them.
  • Save a macro to your config: let @a = 'iHello\<Esc>'. Same data structure as a yanked register — just initialized from Vimscript.

The dual nature of register-as-text and register-as-keystrokes is one of Vim's most elegant design decisions. A clipboard, a search history, a macro store, and a filename cache — all the same primitive.

Common patterns

Pattern Why it works
"0p Paste the last yank, ignoring intervening deletes.
"_dd Delete a line without overwriting the unnamed register.
"+yiw Yank a word to the system clipboard.
"Ayy Append this line to register a (collect lines).
"1p.. Paste deletes 1, 2, 3 in order — by repeating ., the numbered register paste increments.
"/p Insert the last search pattern as text.
Ctrl-R0 (Insert mode) paste the last yank into the buffer you're typing in.
Ctrl-R=1+1Enter (Insert mode) compute and insert. Expression register live.

When to use which

Default: just don't say a register. Yank, delete, and paste with the unnamed register. Most edits are too short-lived to bother.

Holding a clip across deletes: yank into a named register ("ay) so your subsequent deletes don't clobber it.

System clipboard: prefix with "+ to bridge to and from other apps.

Throwaway delete: prefix with "_ when you're deleting junk you don't want to paste later.

Recovering a recent delete: "1p pastes the most recent line-delete; press . repeatedly to walk back through "1, "2, "3, …

Tip. If you find yourself wanting "more clipboards," you already have 26. The hard part isn't capacity — it's remembering what's in "a versus "b. :reg is your friend.
🔧 Under the Hood — Registers are a typed key-value store

Vim's register file is a Map<char, RegisterEntry> where each entry holds (text, type). The keys are characters: lowercase letters, uppercase letters (which append to their lowercase counterpart), digits, and a fistful of special characters. Different command families write to different keys automatically. It's much closer to a multiplexed scratchpad than to a clipboard.
