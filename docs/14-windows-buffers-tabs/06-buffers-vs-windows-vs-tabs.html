Buffer vs Window vs Tab

Three concepts most editors collapse into one. Vim keeps them separate.

id: windows.buffers-vs-windows-vs-tabs · part: windows

A buffer is a file in memory. A window is a viewport onto a buffer. A tab is a layout of windows. Most editors call all three 'tabs' — that's why Vim's model feels strange at first.

Most editors give you one concept — call it a tab, a panel, whatever — and that one thing represents "a file you're editing." Vim breaks it into three. Once you see the split, the rest of Vim's window model snaps into place.

🔧 Under the Hood — The Three Layers

Buffer — a file in memory. You open a file, Vim loads its contents into a buffer. The buffer is the editable text. There's no UI for it directly; you see a buffer through a window.

Window — a viewport onto a buffer. The split layout you see is windows. One buffer can be shown in many windows; one window shows one buffer at a time.

Tab — a layout of windows. A way to save and switch between window arrangements.

Concrete picture: open three files (three buffers). Split the screen into two windows; each window shows a buffer. Open a tab; lay out four windows there showing yet other buffers. Total: 7 buffers, 6 windows, 2 tabs. Closing a window doesn't unload its buffer — the buffer is still in memory, just not currently visible.

🔧 Under the Hood — Why This Matters

Once you treat buffers as your unit of work and windows/tabs as transient layout decisions:

  • You stop trying to "close the tab" and start "closing the window, the file's still there."
  • You can have 50 buffers loaded with no UI clutter — they're invisible until you :b to one.
  • You arrange splits the way that helps your current task, not your file inventory.
  • You stop fighting Vim's defaults.

Your mental model: buffers are the data; windows are the UI; tabs are the saved UI layouts. Most professional Vim users live in 1–3 windows and zero tabs, switching between many buffers.
