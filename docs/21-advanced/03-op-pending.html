Operator-Pending Mode

The fourth mode you didn't know about.

id: advanced.op-pending · part: advanced · keys: d, c, y · lessons: 487, 488

Between pressing an operator (d/c/y/...) and pressing the motion, Vim is in operator-pending mode. It's a real mode with its own status ("-- OPERATOR --") and its own mappings.

Vim has four base modes, not three. Between an operator key like d and the motion that completes it, Vim is in operator-pending mode — waiting for a motion or text object.

🔧 Under the Hood — Why It's a Real Mode

After you press d, Vim enters a state where it accepts only motions and text objects. The status line shows -- OPERATOR -- (in many configurations) or remains blank, depending on your settings. Esc cancels and returns to Normal.

The op-pending state is a real mode for two reasons:

1. Custom mappings. You can define mappings that only apply during operator-pending with onoremap. For example, :onoremap p i( makes dp mean "delete inside parens" — but p alone in Normal mode (which is paste) is unaffected. 2. Motion modifiers like v, V, Ctrl-V change the operator's character/line/block flavor — so dvG forces character-wise delete to end of file even though G is normally line-wise.

This is the mechanism Tim Pope's vim-surround and other plugins use to add custom text objects.

🔧 Under the Hood — The Modifiers v / V / Ctrl-V

After an operator, you can press v, V, or Ctrl-V before the motion to override the motion's natural shape:

  • dVfx — delete linewise from cursor through next x (whole lines, not character range).
  • dvG — delete character-wise to end of file (would normally be linewise).
  • dCtrl-VG5l — delete a block.

Useful when the default "motion shape" doesn't match what you want.

Tip. Try :onoremap il :<C-u>normal! ^v$<CR> — defines a custom "inner line" text object so dil deletes line content without the leading whitespace or trailing newline.
🔧 Under the Hood — Operator-pending is the most important hidden mode

After you press an operator key like d, c, y, >, <, gu, gU, the editor enters a fourth major mode where it waits for a motion or text object to resolve a range. Until that resolves, no buffer change happens. The operator is staged with its count; the resolved motion provides (start, end, type); only then does the operator function execute. This pending state is also why doubled forms (dd) work: the parser sees the operator key while in op-pending and short-circuits to 'current line, linewise.'
