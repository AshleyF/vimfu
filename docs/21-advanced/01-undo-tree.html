Undo as a Tree

g- and g+ walk by time. :undolist shows the branches.

id: advanced.undo-tree · part: advanced · keys: u, Ctrl-R, g-, g+ · lessons: 481, 483

Vim's undo isn't a stack — it's a tree. Every undo branch is preserved. g- and g+ walk by chronological time; :undolist shows all branches; :earlier/:later jump by time intervals.

In most editors, undo is a stack: redo a mistake, the original is gone forever. In Vim, undo is a tree — every branch is kept.

Key Action
u Undo previous change (along current branch)
Ctrl-R Redo (along current branch)
g- Go to OLDER state by chronological time
g+ Go to NEWER state by chronological time
:undolist Show the undo tree
:earlier 5m State as of 5 minutes ago
:later 30s 30 seconds forward
:earlier 10 10 changes back
🔧 Under the Hood — The Tree Structure

Each change (one Insert session, one delete, one substitute, etc.) creates a new node in the undo tree. u walks toward the parent of the current node; Ctrl-R walks back toward the most recent child.

When you edit, undo, then edit again, the new edit becomes a new branch — and the previous redo path is preserved as a sibling. Without g-/g+, you'd never reach that sibling again.

  • g- / g+ walk by time — they ignore the tree shape and visit nodes in chronological creation order. This lets you reach any state regardless of branching.
  • :undolist shows the tree as a table: number, time, save count, branch info. Use :undo N to jump to a specific node.
  • :earlier 5m and :later 30s jump by wall-clock time — priceless when you realize you broke something a few minutes ago.

The undo tree is persisted across sessions if you :set undofile — Vim writes a .<filename>.un~ file beside each saved file. Real time-travel.

Tip. Plugins like undotree visualize the tree as a sidebar. The data is all there in :undolist — the plugin just makes it readable.
